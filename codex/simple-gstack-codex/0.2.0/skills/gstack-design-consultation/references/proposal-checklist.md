# Proposal Checklist

Before writing `DESIGN.md`, make sure the proposal answers these points.

## Product Fit

- Who uses the product?
- What job are they doing?
- What emotional state are they in?
- Is this marketing, app UI, internal tooling, editorial, game-like, or hybrid?
- What should feel familiar because users expect it?
- What should feel distinctive because the product needs a face?

## System Decisions

- Aesthetic direction has a name and one-sentence rationale.
- Typography names specific fonts and roles.
- Colors include hex values, semantic meanings, and contrast intent.
- Spacing defines a scale and density.
- Layout defines information density and grid behavior.
- Components define surfaces, radius, borders, focus, disabled, loading, empty, and error states.
- Motion has timing, easing, and purpose.
- Responsive rules are intentional, not "stack on mobile."
- Accessibility constraints are explicit.

## Safe And Risk

Include at least two `SAFE` decisions and two `RISK` decisions.

`SAFE` means users in the category expect it. `RISK` means the product becomes more memorable but gives something up. Explain both sides.

## Anti-Slop

Reject generic choices unless there is a strong reason:

- default system-font-only identity
- purple-blue gradient brand
- decorative three-card feature grid
- icon circles as personality
- all-centered sections
- vague "unlock the power" copy
- one-note palette
- every surface with the same large radius
